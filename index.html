Back To Front

Lessons Learnt While Building My First Web App

Over the past few weeks, I’ve been building a little app. Before now I’ve only ever designed UIs and built simple HTML, CSS, JS websites with no log-in system and no form of stored data. This new one, which I called Depiction.co has both. (Try it out so you have some idea of what I’m talking about in this post.) It’s a web app that allows each user to log in with Twitter and save a status update with a colour, which are then written to a database with their Twitter username and some metadata. Each user has a profile page which displays their current status and colour. It’s simple, and largely useless, but it was a nice first project to convince me that user management and data storage isn’t as horrific as I first thought.

While building Depiction.co, I learnt quite a lot (mostly due to the help Joe gave me). I’d like to share some of those things with you - it might even convince you that it’s worth making your own website or app.

Because everything I’ve made until now has been design-centric and not based on any back-end or function which I had to code myself, I took a bottom-up approach to the project. First, I designed the posting page in the browser. It was just a static HTML, CSS website. I made sure each element on the page received as much time as it would if I was only designing the UI with no back-end. This, I think, ensured that the result had at least some focus on design. If I hadn’t approached it from front-end to back-end, I’m sure the design wouldn’t be as good.

I then used the little I did know about making dynamic pages and added some JS and jQuery to the page. When the “Update” button was clicked, I set a variable to store the value of the textarea and another to store the selected colour by checking the opacity of the element. At this point, I made sure I could send the data wherever I needed to by popping it up in an alert. I didn’t know how I would send it to a database, or whatever I ended up building, but I had a pretty good idea that this would be a good base for it.

When I first looked at how to build the back-end of Depiction, I approached it quite messily, trying to hack together a service that could pull data from different APIs. I began by using a service I’d heard of a few times, Hull.io, to manage user accounts. I used their Identity component (loaded by JavaScript) to allow users to sign in with Twitter, then a registration form to collect the user’s status and selected colour and Hull’s service to store the data. Up to this point everything had gone well; I had sacrificed a few things in terms of the UI, but the user’s data was saving fine.

The trouble came when I tried to retrieve the data and display it per user. I am sure it’s possible to do, but I couldn’t work out how to use Hull to write the data, even when I looked deeper into their API. I asked a few friends (much more experienced than me) and they couldn’t work it out either. One of them, Joe, suggested that it would be easier to build my own back-end from scratch using PHP. Seeing no way forward with Hull, I took his advice. Trying to take a shortcut by using Hull had only wasted time, and I hadn’t learnt anything new by doing it - a lesson I’ll try to take heed of in the future.

For the next hour or so I learnt the basics of PHP and comfortingly found that it was similar to JS. Following a basic tutorial path, I quickly came across POST and GET requests and realised these would likely form a key part of my code. Joe then showed me a useful library, RedBeanPHP, which can be used to manage databases in PHP. He also wrote a Twitter login PHP file for me and gave me a tonne of advice on fitting everything together and finding bugs.

While working on the back-end I noticed a few things about development in general. When testing and making sure things worked, I had no inclination to make sure the design was good. I was fully focussed on getting things to work. This method worked well in that it showed me when things were and weren’t working with no clutter, but the lack of even basic CSS didn’t encourage me that the finished site would look good. Prior to building Depiction, if I had seen an app or a website that worked well but suffered from having a bad UI or an illogical UX, I had just assumed that the developers didn’t know what they were doing. But now I see it’s not that at all. When you code something it rarely works first time. There are bugs aplenty to find and crush - it takes a great deal of effort to even build the working system that puts content into your design framework. Because of this effort, I found that I felt a real sense of achievement when the code actually worked. The reward now wasn’t seeing a well designed layout but just the plaintext content on the page.

This variation in attitude is the main thing I think keeps designers and developers apart. Designers see content as a part of their design and have to put the system used to fetch it to the back of their mind, thinking only about the user’s interactions with the interface. Developers have to think in terms of inputs and outputs, hence the design isn’t their priority, data is. I think there’s a tremendous value in one understanding what the other has to do and what motivates them. From this simple exercise I feel I have a much better idea of what developers aim to do. Now I know the work that goes in to even the simplest things, like getting a Tweet to display on a screen or a string to save to a database. Of course it’s this same disparity that makes developers and designers both so essential on a project as well. Each mindset is needed to get the best result, the best of both design and function.

So, that’s what building Depiction taught me - not bad for a tiny web app, I’d say. By approaching the project in a back to front fashion (front-end before back-end), I learnt that building on what I already know can make new concepts, even entirely new programming languages, seem easier to grasp. Through making the mistake of trying to take a shortcut using Hull, I found that sometimes it’s more valuable to take the long way round and learn something in the process than try a shortcut and get stuck in the mud. Most importantly, I now know better than before the differences between designers and developers and what drives them - I suspect this will make my approach to design in the future much more compatible with the way devs work. Only time will tell…
